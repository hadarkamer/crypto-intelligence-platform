# TradingView Indicator — Stage 1 Shadow Mode

This integration receives and stores technical-indicator signals. It does **not** change Liquidity Score, Alerts, or Watch ranking.

## Endpoint

`POST /webhooks/tradingview`

Production URL example:

`https://YOUR-RENDER-DOMAIN/webhooks/tradingview`

## Required environment variable

`TRADINGVIEW_WEBHOOK_SECRET=<long random secret>`

The service rejects all TradingView requests when the secret is missing.

## Alert message JSON

Use valid JSON in the TradingView alert message. The first supported contract is:

```json
{
  "secret": "YOUR_SECRET",
  "source": "tradingview",
  "symbol": "{{ticker}}",
  "exchange": "{{exchange}}",
  "timeframe": "{{interval}}",
  "direction": "LONG",
  "technical_score": 78.5,
  "signal_timestamp": "{{timenow}}",
  "bar_close_timestamp": "{{time_close}}",
  "is_confirmed": true,
  "indicator_version": "friend-indicator-v1",
  "settings_profile": "default"
}
```

`direction` and `technical_score` must be generated by the Pine indicator. Do not hard-code `LONG` in the final alert unless the alert itself is long-only.

Supported aliases include:
- `ticker` instead of `symbol`
- `interval` or `tf` instead of `timeframe`
- `score` instead of `technical_score`
- `side` or `signal` instead of `direction`
- BUY/BULLISH → LONG; SELL/BEARISH → SHORT; FLAT → NEUTRAL

## Validation

- Symbol is normalized, e.g. `BINANCE:BTCUSDT` → `BTC` with exchange `BINANCE`.
- Score must be between 0 and 100.
- Direction must normalize to LONG, SHORT, or NEUTRAL.
- Timeframe is stored explicitly.
- Duplicate payloads receive HTTP 200 but are not inserted twice.
- Both normalized data and the original payload are stored.

## Verification

- `GET /technical/status` returns the latest 20 stored signals.
- Telegram command `/technical_status` returns the latest 10 signals.
- Server logs show `stored`, `duplicate`, or a validation error.

## Stage-1 acceptance checks

1. Send one BTC 15m test alert and confirm `stored: true`.
2. Resend the exact payload and confirm `duplicate: true`.
3. Send BTC 1h and confirm it remains separate from BTC 15m.
4. Send `BINANCE:ETHUSDT` and confirm it is stored as symbol `ETH` and exchange `BINANCE`.
5. Send an invalid score such as 140 and confirm HTTP 422.
6. Confirm timestamps are UTC ISO-8601 in `/technical/status`.
7. Keep this in Shadow Mode until the actual indicator payload and score semantics are documented.
